## Reference: INC-042 class — API connection pool exhaustion (NOT database overload)

Symptoms:
- HTTP 503 or high latency on api-server; alerts on p99 / error rate.
- App logs: "Cannot acquire connection from pool", "HikariPool-1 — Connection is not available, request timed out" (or equivalent pool vendor).
- Kubernetes: pods often still Running/Ready (misleading).

Red herrings (common wasted time):
- PostgreSQL looks healthy: low CPU, few active connections vs max — the app pool can still be exhausted with pool size 1 while DB has spare capacity.
- No recent deployment — root cause may be ConfigMap/Secret/env change, not a new image.
- Traffic "normal" — even normal concurrency exceeds a pool of size 1.

Correct investigation order:
1. Read app logs for pool timeout / acquire timeout messages.
2. Check application DB pool SIZE configuration (e.g. Hikari maximum-pool-size, DB_POOL_SIZE in ConfigMap/env), not only DB server connection count.
3. Compare against known-good values and recent platform/cost-optimization changes.

Example fix (Kubernetes):
- kubectl get configmap api-config -n production -o yaml  # verify DB_POOL_SIZE
- kubectl patch configmap api-config -n production -p '{"data":{"DB_POOL_SIZE":"20"}}'
- kubectl rollout restart deployment/api-server -n production
- Verify new pods: env | grep DB_POOL

Root cause category: configuration drift (pool too small).

Post-incident: alert on dangerously low pool size; require review for ConfigMap changes in prod; document cost-optimization rollbacks.

When the user asks a question, give: likely root cause, why DB looked fine, exact kubectl checks, fix commands, and 1–2 preventive action items. Prefer concise structured output (headings + bullet lists).
